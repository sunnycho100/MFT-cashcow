## Mission

Build a system that:
1. Learns from historical + newly collected market data.
2. Improves over time without overfitting itself to death.

## What We Will Do (Plan)

## Phase 0 — Safety + correctness before live money

- Fix code correctness blockers (e.g., paper-trade signal mapping inconsistency).
- Standardize one source of truth for labels, class mapping, confidence, and side logic.
- Enforce non-negotiable risk kill-switches:
  - Global max daily loss.
  - Max drawdown halt.
  - Per-trade risk cap.
  - Kraken dead-man switch heartbeat.

## Phase 1 — Data engine that can keep improving

- Keep existing OHLCV pipeline, then add higher-signal features incrementally:
  - Kraken order book microstructure snapshots.
  - Trade flow features.
  - (Optional) futures/funding/open-interest features if futures route is enabled.
- Build a feature registry with feature lineage + availability times (prevent leakage).
- Persist all training/backtest/live inference datasets with version tags.

## Phase 2 — Model architecture for “high-confidence trading”

- Separate model responsibilities:
  - Regime model: trend/chop/risk-on-risk-off.
  - Direction/edge model for long opportunities.
  - Separate short model (not inverse-long proxy).
- Add abstention model/policy:
  - If confidence not high enough, do nothing.
- Calibrate probabilities (isotonic/sigmoid) before using thresholds for trade decisions.

## Phase 3 — “Short only when 100% sure” policy (practical translation)

Absolute 100% certainty is not possible in markets.  
Implementation policy:
- Require all short gates simultaneously:
  1. Short model calibrated probability >= strict threshold.
  2. Regime confirms downside structure.
  3. Liquidity/slippage and spread checks pass.
  4. No conflict with higher-timeframe filter.
  5. Risk budget available and drawdown guard not active.
- If any gate fails -> no short trade.

## Phase 4 — Validation that resists overfitting

- Walk-forward evaluation with purge/gap where needed.
- Weekly champion/challenger:
  - Champion stays live.
  - Challenger trains on latest data.
  - Promote challenger only if robust OOS metrics beat champion after costs/slippage.
- Track:
  - Net expectancy.
  - Sharpe/Sortino.
  - Max DD.
  - Stability by regime.
  - Trade calibration quality.

## Phase 5 — Kraken execution stack (production)

- Build execution service around Kraken private APIs:
  - Auth + token lifecycle.
  - Order router.
  - Fill reconciler.
  - Position/account sync.
- Use `validate=true` preflight for order checks when appropriate (WS v2 add_order).
- Use reduce-only semantics for close-only logic where supported.
- Keep cancel-all-after heartbeat running continuously.
- Start in paper/sim mode, then micro-size live rollout.

## Weekly Operating Cadence (Forever-Improving Loop)

1. Monday: retrain challengers with latest data snapshot.
2. Tuesday: run walk-forward + stress/backtest suite.
3. Wednesday: calibration + threshold sweep (long and short separately).
4. Thursday: paper-forward shadow run against champion.
5. Friday: promote only if improvement survives cost/risk gates.
6. Daily: drift monitor and risk monitor; auto-disable trading on breach.

## March 12, 2026 Research Addendum — What Actually Matters

## What `v2` currently gets right

- It already moved away from the failed `v1` deep-model stack toward simpler, more inspectable models.
- It already discovered that **trend structure matters** more than just asking a model to predict the next move from candles.
- It already treats ML as a probability or filter in several test files instead of trusting raw directional predictions blindly.

## What matters most in quant trading

The research points to five priorities:

1. **Edge source beats model complexity.**
   - General ML finance research shows ML helps when it captures nonlinear interactions, but also highlights low signal-to-noise and overfit risk.  
     Source: Gu, Kelly, Xiu — NBER / RFS  
     <https://www.nber.org/papers/w25398>  
     <https://academic.oup.com/rfs/article-abstract/33/5/2223/5758276>

2. **Crypto-specific factors matter more than generic macro factors.**
   - Crypto returns are predicted more by crypto-specific signals than by standard stock / macro factors, with strong evidence for momentum and investor attention.  
     Source: Liu & Tsyvinski — NBER / RFS  
     <https://www.nber.org/papers/w24877>

3. **Order flow / microstructure is closer to the causal mechanism.**
   - Order flow directly drives price formation and has been shown useful for Bitcoin directional modeling.  
     Source: Westray, Deep Recurrent Modelling of Stationary Bitcoin Price Formation Using the Order Flow  
     <https://arxiv.org/abs/2004.01499>

4. **Trend and volume information do have predictive value, but horizon matters.**
   - Daily/weekly price-based trend indicators and longer-horizon volume indicators show predictive power in crypto, and ML can improve them.  
     Source: Tan & Tao, *Trend-based forecast of cryptocurrency returns*  
     <https://www.sciencedirect.com/science/article/pii/S0264999323001359>

5. **Execution, costs, and validation are not optional.**
   - Kraken exposes real-time L2 order book, trades, and futures ticker data; if we ignore spread, depth, and slippage, backtests will lie.  
     Sources: Kraken docs  
     <https://docs.kraken.com/api/docs/websocket-v2/book/>  
     <https://docs.kraken.com/api/docs/websocket-v2/trade/>  
     <https://docs.kraken.com/api/docs/futures-api/websocket/ticker/>

## Should we use only ML?

**No.**

Best recommendation:
- Use a **hybrid quant stack**.
- Let deterministic strategy logic define the broad playbook.
- Let ML decide:
  - regime,
  - trade ranking,
  - confidence / abstention,
  - maybe entry timing.

Inference from the repo + research:
- `v2`'s strongest direction already points this way.
- The academic evidence does **not** support "just throw a bigger model at 1H candles."
- Shallow / structured ML often beats deeper models in noisy financial settings because data is limited and the signal is weak.  
  Source: Gu, Kelly, Xiu  
  <https://academic.oup.com/rfs/article-abstract/33/5/2223/5758276>

## My recommendation for the actual system

### Core system

- **Core strategy:** breakout / trend-following + regime filter.
- **ML role:** regime classifier + trade quality scorer + short gate.
- **Execution role:** trade only when edge survives cost and liquidity checks.

### Why

- This matches both:
  - what `v2` is already drifting toward,
  - and what the literature says is actually robust in crypto.

## Data Source Ranking — What We Should Add First

## Tier 1 — Add first

### A) Kraken market microstructure

Use Kraken-native:
- L2 order book depth,
- top-of-book spread,
- trade prints,
- short-term imbalance / aggressor flow.

Why:
- This is directly tied to execution quality and short-horizon price pressure.
- Kraken officially provides:
  - L2 book stream,
  - recent trade stream,
  - futures ticker with funding fields for perpetual products on non-US derivatives feeds.

Sources:
- <https://docs.kraken.com/api/docs/websocket-v2/book/>
- <https://docs.kraken.com/api/docs/websocket-v2/trade/>
- <https://docs.kraken.com/api/docs/futures-api/websocket/ticker/>

### B) Cross-exchange price / basis / premium

Examples:
- Kraken vs Coinbase premium,
- Kraken vs Binance basis,
- spot vs futures basis.

Why:
- This captures dislocations, regional demand, and short-term lead-lag.
- It is more "market-structure" than "social noise."

### C) Derivatives state

Examples:
- funding rate,
- funding rate change,
- open interest change,
- basis term structure,
- liquidation pressure if the source is trustworthy.

Why:
- These are crypto-native positioning indicators.
- Even when execution is on Kraken, derivatives positioning on major venues can contain useful information about crowd leverage and squeeze risk.

Important US note:
- As of **December 5, 2025**, Kraken says non-US Kraken Derivatives supports perpetuals, while **Kraken Derivatives US** offers US crypto futures from CME.  
  Source: <https://support.kraken.com/hc/en-us/articles/360023786632-Kraken-Futures-eligibility>
- As of **December 10, 2025**, Kraken says US margin trading remains restricted to clients meeting specific ECP requirements.  
  Source: <https://support.kraken.com/articles/360061972272-margin-trading-and-eligible-contract-participant-ecp-self-certification-for-u-s-clients>

Implication:
- If the trading account is US-based, we should design shorting and leverage assumptions around **actual Kraken eligibility**, not generic crypto exchange assumptions.

### D) Multi-timeframe trend / volume

Keep and improve:
- 4H / daily trend regime,
- 1H setup quality,
- 15m entry timing,
- time-of-day / session effects.

Supporting research:
- Crypto trend indicators can predict returns across horizons, and ML can improve them.  
  <https://www.sciencedirect.com/science/article/pii/S0264999323001359>
- Bitcoin studies find no persistent return anomalies across clock-time buckets, but persistent **volume** patterns do exist.  
  <https://www.sciencedirect.com/science/article/abs/pii/S1544612319301710>

Practical conclusion:
- Time features are useful as context features, but not as a standalone edge.

## Tier 2 — Add after Tier 1

### E) On-chain / crypto flow features

Examples:
- exchange inflows/outflows,
- stablecoin net issuance / inflows,
- whale transfer activity,
- ETF flow proxies where relevant.

Why:

- They are slower but often more structural than social chatter.

## Tier 3 — Optional / experimental only

### F) Polymarket API

What it is good for:
- event probabilities,
- macro narrative shifts,
- crowded expectations about discrete catalysts.

What the docs show:
- public order book and WebSocket data are available,
- market/event metadata is easy to fetch by slug,
- rate limits are generous for public data.

Sources:
- <https://docs.polymarket.com/concepts/markets-events>
- <https://docs.polymarket.com/trading/orderbook>
- <https://docs.polymarket.com/api-reference/rate-limits>

Why I would **not** make it core for BTC direction:
- It is strongest for **event-driven forecasting**, not continuous 24/7 BTC micro-direction.
- Prediction markets are reasonably calibrated close to expiry, but can be biased further out.  
  Source: Page & Clemen, *Do Prediction Markets Produce Well-Calibrated Probability Forecasts?*  
  <https://academic.oup.com/ej/article/123/568/491/5079498>
- Polymarket itself restricts order placement from the US.  
  Source: <https://docs.polymarket.com/api-reference/geoblock>

My recommendation:
- Use Polymarket only later, as a **macro/event overlay**:
  - Fed,
  - election,
  - ETF approval odds,
  - recession / inflation event proxies,
  - possibly BTC target / milestone markets if liquid enough.
- Do **not** use it as a primary fast-trading signal.

### G) Reddit / social word frequency

What the research says:
- Some studies find sentiment / attention can help at daily horizons.
- Other evidence suggests sentiment often reacts to price rather than leads it.

Support for possible value:
- Social / search attention helped forecast crypto returns in some studies.  
  <https://www.nber.org/papers/w24877>  
  <https://arxiv.org/abs/2210.00883>

Support for caution:
- A 2026 paper found the Crypto Fear & Greed Index mostly reacts to price and adds little to next-day Bitcoin predictability.  
  <https://www.sciencedirect.com/science/article/pii/S305070062600006X>

Operational / legal caution:
- Reddit requires approved API usage with OAuth and free-tier rate limits around 100 QPM.  
  <https://support.reddithelp.com/hc/en-us/articles/16160319875092-Reddit-Data-API-Wiki>
- Reddit Data API Terms prohibit using User Content to train an ML / AI model without express permission of rightsholders.  
  <https://redditinc.com/policies/data-api-terms>

My recommendation:
- Do **not** start with raw Reddit word frequency.
- If we use Reddit later, use it only as an **experimental, low-weight attention feature**, not as a core signal.
- Better text sources than raw Reddit counts:
  - finance news headlines,
  - ETF / macro headlines,
  - curated crypto news,
  - maybe a supervised text model tied directly to return prediction.

This is consistent with finance text research showing that text is more useful when the sentiment extraction is **trained for return prediction**, not when it is a generic keyword counter.  
Source: <https://www.nber.org/papers/w26186>

## Bottom-Line Recommendation

If the goal is to build something that can actually survive contact with the market:

- **Do not use ML alone.**
- **Do not start with Reddit word counts.**
- **Do not make Polymarket a core BTC direction signal.**

Instead:

1. Build a **hybrid system** with trend/regime logic first.
2. Add **Kraken microstructure** and **cross-exchange / derivatives** data next.
3. Use ML for:
   - regime detection,
   - trade ranking,
   - calibrated confidence,
   - short permissioning.
4. Treat Polymarket and Reddit as later-stage auxiliary features, not primary alpha.
